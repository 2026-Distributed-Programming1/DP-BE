# 웹 사이트 구상 및 유스케이스 차이 검토

> 기준 문서: `Usecase_scenario.md`
> 현재 상태: REST API 전환과 배포는 완료. 프론트 연동 관점 도메인별 1차 검토는 `FrontendIntegrationReview.md`에 정리됨.
> 원칙: 이 문서는 웹 사이트를 어떻게 구성하면 좋을지와, 기존 유스케이스 시나리오 대비 현재 API/구상 차이를 기록한다. 코드 수정은 별도 승인 후 진행한다.

## 1. 내가 보는 웹 사이트 방향

이 프로젝트는 단일 공개 웹사이트라기보다 **보험 업무 운영 포털 + 고객 셀프서비스**에 가깝다.

권장 구조는 다음과 같다.

1. **고객 포털**
   - 상담 신청
   - 가입 보험 조회
   - 보험금 청구
   - 사고 접수
   - 보험료 납입
   - 계약 해지 신청
   - 문의/FAQ/문의 내역 조회

2. **직원 업무 포털**
   - 계약 관리
   - 만기 계약 관리
   - 보험료 납부 내역 관리
   - 해약 환급 산출/지급
   - 사고/청구/조사/산출/지급 관리
   - 상담/면담/제안/청약/인수심사 관리
   - 영업활동/채널/성과급 관리
   - 교육 계획/제반/실행 관리
   - 고객 등록/조회

3. **관리자/공통 영역**
   - 로그인/권한
   - 파일/이미지 관리
   - 통계/리포트
   - 코드/옵션 관리

현재 API만 놓고 보면 직원 업무 포털 쪽 CRUD와 상태 전이 API는 어느 정도 만들어져 있다. 고객 직접 회원가입과 세션 로그인은 1차 구현됐지만, 고객 포털용 조회/신청 API, 파일 접근, 통계/그래프, 직원 세부 권한은 아직 약하다.

알림 표시, 화면 이동, 버튼 노출, 토스트 메시지, 업무 단계 시각화는 프론트 책임으로 둔다. 백엔드는 화면을 직접 지시하기보다 상태값, 업무번호, 날짜, 검증 결과를 안정적으로 제공하는 데 집중한다.

## 3. 유스케이스 대비 큰 차이

### 3.1 유스케이스는 화면 중심, 현재 API는 엔티티 중심

`Usecase_scenario.md`는 “화면에서 사용자가 어떤 버튼을 누르고 어떤 패널/팝업이 뜨는가”를 중심으로 작성되어 있다.

현재 API는 대부분 다음과 같은 구조다.

- 목록 조회
- 상세 조회
- 생성
- 승인/거절/확정/실행 같은 상태 전이

즉 업무 데이터는 다룰 수 있지만, 유스케이스의 화면 UX는 API가 아니라 프론트에서 조합해야 한다.

- 현재 단계
- 다음 액션
- 버튼 노출 가능 여부
- 완료 후 이동할 화면
- 알림 발송 결과
- 화면용 집계/요약

이 차이는 크지만 치명적이지는 않다. 현재 방향은 프론트가 상태를 해석하고 화면 흐름을 담당하는 것이다. 백엔드는 화면용 `nextAction`, `canApprove` 같은 UI 파생 필드를 필수 작업으로 두지 않고, 대신 상태값과 상태 전이 규칙을 문서/명세로 명확히 제공하는 편이 맞다.

### 3.2 유스케이스는 역할이 명확하지만 현재 API는 권한이 없음

시나리오에는 고객, 판매채널, 영업 관리자, 영업교육 담당자, 계약관리 담당자, 보상 담당자 같은 역할이 나온다.

현재 API에는 HTTP 세션 로그인과 기본 role 권한이 1차로 들어갔다. 다만 유스케이스에 나오는 세부 직원 역할까지는 아직 분리되지 않았다.

차이:

- 유스케이스: 역할별 메뉴와 처리 권한이 전제됨
- 현재 API: 로그인 세션과 `CUSTOMER` 소유권 검증은 있으나, 직원 도메인별 권한은 아직 기본 `STAFF` 중심

보완:

- HTTP 세션 로그인과 기본 role(`CUSTOMER`, `STAFF`, `ADMIN`)은 1차 구현 완료
- 고객 직접 회원가입은 `POST /api/auth/signup/customer`로 1차 구현 완료
- `CUSTOMER`의 주요 고객 데이터 소유권 검증은 서비스 계층에서 처리
- 직원 세부 역할은 `CONTRACT_STAFF`, `CLAIM_STAFF`, `UNDERWRITING_STAFF`, `SALES_STAFF`, `EDUCATION_STAFF`, `FINANCE_STAFF`, `DISPATCH_STAFF`로 나누는 방향
- 1차 직원 계정은 actor 테이블과 연결하지 않고 `auth_users.role`만으로 API 접근 권한을 판단
- 직원 actor 연결은 처리자 기록, 담당자 배정, 담당자별 실적/한도/지역/교육 이력이 필요해지는 시점에 `linked_actor_type`, `linked_actor_id` 같은 확장으로 검토
- 메뉴 노출과 API 접근 제어를 같은 역할 체계로 맞춰야 함

### 3.3 유스케이스는 알림을 자주 요구하지만 현재 범위에서는 프론트 책임임

시나리오에는 다음 알림이 반복된다.

- 영업 관리자에게 교육 계획 승인 요청 알림
- 고객에게 면담 일정 알림
- 인사 담당자에게 채널 등록 요청 알림
- 고객에게 문의 접수 문자
- 계약관리 담당자에게 만기 응답 알림
- 판매채널에게 교육 수료 알림

현재 결정은 알림 표시와 화면 메시지를 프론트 책임으로 두는 것이다. 따라서 위 항목은 백엔드 결함이라기보다 프론트 UX 설계 항목이다.

보완:

- 1차는 토스트, 배지, 리스트 강조, 화면 이동 안내를 프론트에서 처리
- 백엔드는 화면 알림용 API를 만들지 않음
- 실제 SMS/email/push 같은 외부 메시징이 필요해지는 경우에만 별도 기능으로 재검토

### 3.4 유스케이스는 파일 업로드/다운로드를 포함하지만 현재 파일 정책은 부분적임

현재 파일 관련 상태:

- 출동 사진은 multipart 업로드 가능
- 출동 사진 응답은 파일명만 제공
- 문의 첨부는 파일명/크기 메타데이터만 받음
- 실제 다운로드 URL, S3 object key, signed URL 정책은 없음

차이:

- 유스케이스: 첨부 파일을 실제 선택하고 상세에서 확인
- 현재 API: 일부는 메타데이터, 일부는 로컬 저장만 가능

보완:

- S3 저장 기능 설계
- 파일 메타데이터 테이블/응답 정리
- 업로드 성공 후 `fileUrl` 또는 `objectKey` 제공
- 이미지 미리보기와 첨부 다운로드 정책 분리

### 3.5 유스케이스는 통계/그래프/엑셀을 요구하지만 현재 API는 제한적임

특히 계약 통계 유스케이스는 다음을 요구한다.

- 납부 현황 기간 필터
- 청구 이력 테이블
- 계약 유지율 그래프
- 전체 계약 통계
- 월별 신규 계약 추이
- 보험 종류별 계약 비율
- 보험료 수납 현황 요약
- 엑셀 다운로드

현재 API는 `GET /api/contract-statistics`, `POST /api/contract-statistics`, `GET /api/contract-statistics/history` 수준이다.

차이:

- 유스케이스: 분석 대시보드
- 현재 API: 단순 통계 snapshot

보완:

- 통계 전용 read model 또는 query API 필요
- 차트용 series 응답 필요
- 엑셀 다운로드 endpoint 필요

### 3.6 유스케이스는 검색 필터가 풍부하지만 현재 API 필터는 일부만 있음

많은 화면에서 기간, 상태, 이름, 고객번호, 계약번호, 보험종류, 채널유형 필터를 요구한다.

현재는 일부만 구현되어 있다.

예:

- 계약 목록: type/page/size 중심
- 영업활동/평가: 기간, 채널유형, page/size 있음
- 문의: customerName/status 있음
- 교육/청구/상담/면담 상당수: 배열 전체 반환

보완:

- 테이블 화면부터 `page/size/total/items` 정책 적용
- 기간/status/customerName/type 등 공통 필터 추가
- 작은 옵션 목록은 배열 반환 유지 가능

## 4. 도메인별 웹 화면 구상과 차이

### 4.1 영업

시나리오 화면:

- 영업 활동 관리
- 판매채널 모집
- 판매채널 채용 심사
- 영업조직 평가
- 성과급 지급 요청
- 고객 정보 등록
- 활동 계획 작성

현재 API로 가능한 것:

- 영업활동 목록/등록
- 채널 모집 목록/등록
- 채널 심사 목록/등록/승인/거절
- 영업조직 평가 목록/등록
- 성과급 요청 생성
- 고객 등록 목록/생성
- 활동 계획 목록/상세/생성

차이:

- 성과급 요청 목록/상세 없음
- 영업활동의 “개선 지시 등록”이 별도 개념으로 분리되어 있지 않음
- 채널 모집에서 영업활동 화면으로 복귀하는 화면 흐름은 프론트 라우팅 책임
- 채널 심사 승인 후 인사 담당자 알림은 프론트 화면 메시지/업무 목록 강조로 처리
- 고객 등록이 공통 고객 master 생성인지 영업 등록 이력인지 불명확
- 활동 계획 검토/확정 흐름이 유스케이스 대비 단순함

내 구상:

- 영업은 직원 포털의 별도 상위 메뉴로 둔다.
- 영업활동/평가/활동계획은 테이블 + 우측 상세 패널 구조가 적합하다.
- 성과급 요청은 영업조직 평가 상세 패널 안의 하위 action으로 두는 것이 자연스럽다.

보완 우선순위:

1. 성과급 요청 목록/상세
2. 평가-성과급 요청 연결 검증
3. 고객 등록과 공통 고객 CRUD 관계 정리
4. 활동계획 승인/검토 상태 전이 추가 여부 결정

### 4.2 교육

시나리오 화면:

- 교육 계획안 작성
- 교육 계획 승인/반려
- 교육 제반 등록
- 교육 진행

현재 API로 가능한 것:

- 교육 계획 목록/상세/생성/승인/반려
- 교육 제반 목록/상세/생성
- 교육 실행 목록/상세/생성

차이:

- 교육 제반 등록 시 승인 상태 검증 부족
- 교육 대상자 명단 첨부 파일은 실제 파일 업로드가 아니라 문자열 목록 중심
- 교육 완료 후 판매채널 수료 알림은 프론트 화면 메시지/업무 목록 강조로 처리
- 제반/실행 응답에 원본 계획 요약이 부족

내 구상:

- 교육은 `계획 -> 승인 -> 제반 -> 실행` 단계형 workflow 화면이 적합하다.
- 교육 계획 상세에서 제반 등록 버튼을 노출하고, 제반 상세에서 교육 실행 버튼을 노출한다.

보완 우선순위:

1. 승인된 계획만 제반 등록 허용
2. 실행 중복 방지 정책
3. 제반/실행 응답에 교육 계획 요약 추가
4. 대상자 명단 파일 업로드 정책 결정

### 4.3 상담/청약/인수심사

시나리오 화면:

- 상담 요청
- 면담 일정 관리
- 면담 기록 관리
- 보험상품 제안
- 청약서 작성
- 보험상품 조회
- 보험 신청
- 부활 요청
- 인수 심사
- 심사 결과 전달

현재 API로 가능한 것:

- 상담 목록/상세/생성/수락
- 면담 일정 목록/상세/생성/수정/취소
- 면담 기록 목록/상세/생성/수정
- 보험상품 목록
- 제안 목록/생성
- 청약/보험신청/부활 생성
- 인수심사 pending 목록/완료

차이:

- 청약/보험신청/부활 목록/상세 없음
- 심사 결과 전달 알림은 프론트 화면 메시지/상태 표시로 처리
- pending applicationType이 한글 문자열
- 인수심사 결과와 원본 신청 상태 연결 보장이 약함
- 면담 일정과 면담 기록이 직접 연결되지 않음
- 고객/상품 선택이 이름 기반인 곳이 있음

내 구상:

- 상담/청약은 판매채널 업무 탭으로 묶는다.
- 인수심사는 별도 심사 담당자 queue 화면으로 둔다.
- pending 목록은 “심사 대기함”으로 보여주고, 청약/보험신청을 같은 테이블에서 구분한다.

보완 우선순위:

1. 청약/보험신청/부활 목록/상세 API
2. pending type code/label 분리
3. 원본 신청 상태 갱신 보장
4. 면담 일정-기록 연결

### 4.4 사고/청구/보상

시나리오 화면:

- 사고 접수
- 보험금 요청
- 현장 출동 정보 기록
- 손해 조사
- 보험금 산출
- 보험금 지급

현재 API로 가능한 것:

- 사고 목록/상세/생성
- 출동 목록/출동 기록 생성/조회
- 청구 목록/상세/생성
- 조사 생성/조회
- 산출 생성/조회/승인
- 지급 생성/조회/실행

차이:

- 사고와 청구가 직접 연결되지 않음
- 청구 목록에는 원천 상태 데이터만 있고, 현재 단계/다음 액션 표시는 프론트에서 계산해야 함
- 출동 사진은 파일명만 제공
- 예약 지급 실행 조건이 느슨함
- 지급 endpoint namespace가 보험료 납입 payment와 혼동 가능

내 구상:

- 보상 담당자 화면은 “청구 처리 보드” 형태가 좋다.
- 각 청구를 접수/조사/산출/지급 컬럼 또는 단계별 tab으로 표시한다.
- 고객 화면에서는 내 청구 진행 상태만 단순하게 보여준다.

보완 우선순위:

1. 사고-청구 연결 정책 결정
2. 출동 사진 URL/S3 정책
3. 지급 endpoint namespace 정리
4. 청구 상태 전이 규칙 문서화

### 4.5 계약/납입/환급

시나리오 화면:

- 가입 보험 조회
- 보험 해지
- 보험료 납입
- 보험료 납부 내역 관리
- 해약 환급 내역 조회
- 해약 환급금 산출
- 환급금 지급
- 계약 정보 조회
- 계약 통계 정보 관리
- 만기 계약 관리

현재 API로 가능한 것:

- 계약 목록/상세
- 해지 생성/목록/상세
- 납입 가능 계약 조회
- 납입 preview/submit
- 납부 내역 목록/확정/반려
- 환급 산출/목록/상세/확정
- 환급 지급 목록/상세/실행
- 만기 계약 목록/안내 생성/응답 기록
- 계약 통계 snapshot

차이:

- 고객용 “가입 보험 조회”를 위한 고객 인증/내 계약 조회 부족
- 계약 검색 필터가 유스케이스보다 단순함
- 환급/납부/해지 목록의 pagination 부족
- 만기 갱신 처리 자체는 부족함
- 계약 통계/그래프/엑셀 다운로드는 유스케이스 대비 크게 부족함

내 구상:

- 계약 담당자 화면은 계약 테이블 + 상세 패널 + 만기/통계 action으로 둔다.
- 고객 포털은 내 계약 목록/상세/납입/해지 신청으로 단순화한다.

보완 우선순위:

1. 고객 검색/상세 및 고객별 계약 조회 안정화
2. 계약 목록 필터 확장
3. 해지/환급 상태 전이 규칙 문서화
4. 계약 통계는 별도 대시보드 범위로 분리

### 4.6 문의/고객센터

시나리오 화면:

- 1:1 문의
- FAQ
- 문의 내역 조회
- 첨부 파일
- 답변 완료 문의 확인

현재 API로 가능한 것:

- 문의 목록/상세/생성
- 문의 답변
- customerName/status 필터

차이:

- FAQ API 없음
- 실제 파일 업로드/다운로드 없음
- 고객 본인 문의만 조회하는 인증 기반 흐름 없음
- 문의 목록 pagination 없음
- inquiryType enum 오류 처리 보강 필요

내 구상:

- 고객센터는 고객 포털의 독립 메뉴로 둔다.
- FAQ는 static data 또는 별도 FAQ API로 시작할 수 있다.
- 문의 내역은 인증 도입 전에는 customerName 필터로 임시 처리 가능하지만, 운영에서는 고객 인증 기반이어야 한다.

보완 우선순위:

1. 문의 목록 pagination
2. FAQ 데이터/API
3. 문의 첨부 파일 업로드
4. 고객 인증 기반 내 문의 조회

## 5. 내 구상과 기존 유스케이스의 차이 정도

전체적으로 방향은 크게 다르지 않다. 기존 유스케이스는 보험 업무 전반을 잘 나눠놨고, 현재 API도 그 유스케이스를 도메인별로 상당 부분 반영하고 있다.

다만 차이는 세 가지다.

### 차이 1: 기존 유스케이스는 “완성된 업무 시스템”에 가깝다

기존 시나리오는 알림, 문자 발송, 엑셀 다운로드, 통계 그래프, 첨부 파일, 권한, 화면 복귀까지 포함한다.

현재 API는 그중 핵심 데이터 저장과 상태 전이를 먼저 구현한 상태다.

다만 알림 표시와 화면 복귀는 프론트 책임으로 정했으므로 백엔드 보완 범위에서 제외한다. 백엔드가 계속 책임져야 하는 것은 파일 접근, 통계/엑셀, 인증/권한, 데이터 검증, 목록 조회 안정성이다.

차이 정도: 큼.

하지만 이건 설계 방향이 다른 것이 아니라 구현 범위가 아직 덜 찬 것이다.

### 차이 2: 나는 화면을 role 기반 포털로 더 명확히 나누는 쪽이 좋다고 본다

기존 시나리오는 업무별 메뉴가 쭉 나열되어 있다.

나는 이를 다음처럼 나누는 편이 좋다고 본다.

- 고객 포털
- 판매채널 포털
- 계약/보상/교육/영업 관리자 포털
- 공통 관리자 영역

차이 정도: 중간.

이 차이는 API 변경보다 프론트 라우팅/레이아웃 설계에 가깝다.

### 차이 3: 기존 유스케이스는 화면 간 이동을 문서 링크처럼 표현하지만, 실제 웹에서는 업무 queue와 상세 패널 중심이 더 자연스럽다

예를 들어 “성과급 지급을 요청한다”는 독립 메뉴라기보다 영업조직 평가 상세에서 열리는 action modal이 더 자연스럽다.

“교육을 진행한다”도 독립 메뉴라기보다 교육 제반 상세에서 이어지는 다음 단계 action이 더 자연스럽다.

차이 정도: 중간.

백엔드 API는 크게 바꾸지 않아도 되지만, 프론트 화면 설계는 workflow 중심으로 재구성하는 것이 좋다. 백엔드는 이 흐름을 지시하지 않고 상태값과 조회 API를 제공한다.

## 6. 실제 웹 구현 우선순위

프론트를 붙인다면 아래 순서가 현실적이다.

### Phase 1: 내부 직원용 MVP

목표: 프론트가 실제 API를 붙여 업무 화면을 탐색할 수 있게 한다.

- 고객 검색/상세
- enum/options
- 공통 에러 응답
- 계약 목록/상세
- 청구 목록/상세
- 영업활동/평가 목록
- 교육 계획 목록
- 문의 목록

### Phase 2: 업무 workflow MVP

목표: 버튼을 눌러 실제 업무 상태가 이어지게 한다.

- 납입 preview/submit/확정
- 청구 조사/산출/지급
- 교육 계획 승인/제반/실행
- 상담 수락/면담 일정/기록
- 영업조직 평가/성과급 요청

### Phase 3: 고객 포털

목표: 고객이 직접 신청/조회하는 흐름을 만든다.

- 상담 신청
- 내 계약 조회
- 보험료 납입
- 사고 접수
- 보험금 청구
- 문의 등록/내역

### Phase 4: 운영 기능

목표: 유스케이스의 완성도를 맞춘다.

- 로그인/권한
  - DB 스키마 변경 관리는 Flyway로 도입했다.
  - 1차는 HTTP 환경에서 세션 기반 로그인으로 구현했다.
  - Spring Session JDBC 테이블과 사용자/역할 테이블은 Flyway migration으로 추가했다.
  - role은 도메인 엔터티가 아니라 `auth_users.role`에서 관리한다.
  - `Customer`는 보험 업무의 고객 master이고, `AuthUser`는 로그인 계정이므로 분리한다.
  - 고객 계정은 관리자가 기존 고객번호 기준으로 발급하고, 최초 로그인 후 비밀번호 변경을 강제한다.
  - 고객 데이터 소유권은 서비스 계층에서 현재 로그인 사용자의 `linked_customer_id` 기준으로 검증한다.
  - 프론트와 백엔드는 다른 origin이므로 CORS credentials 설정이 필요하다.
  - 세션 쿠키가 브라우저 정책 때문에 안정적으로 동작하지 않으면 JWT 기반 로그인으로 전환한다.
- 파일/S3
- 통계/그래프
- 엑셀 다운로드
- audit/logging

## 7. 결론

기존 유스케이스와 현재 내가 보는 웹 사이트 구상은 큰 방향에서는 같다.

다만 현재 API와 바로 연결 가능한 웹은 “완성된 보험 업무 시스템”이라기보다 **직원용 업무 처리 MVP**에 가깝다.

유스케이스 수준의 완성형 웹으로 가려면 다음 축을 보강해야 한다.

1. 고객/권한/역할
2. enum 값/상태/에러 같은 프론트 공통 기반
3. 상태 전이 규칙 문서화
4. 목록 pagination/filter
5. 파일/S3
6. 통계/리포트

따라서 다음 작업은 `FrontendIntegrationReview.md`의 Batch A와 맞춰서 진행하는 것이 좋다.

- 고객 검색/상세 API
- enum 값 명세
- 공통 에러 응답 보강

이 세 가지를 먼저 끝내면 프론트가 화면을 붙이기 시작할 수 있다.
